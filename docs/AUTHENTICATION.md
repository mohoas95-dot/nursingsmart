# سامانه احراز هویت و بازیابی رمز

## راه‌اندازی

1. یک PostgreSQL مستقل برای هر محیط بسازید.
2. متغیرهای `.env.example` را تنظیم کنید.
3. Client و Migration را اجرا کنید:

```bash
npm run db:generate
npm run db:migrate
npm run db:seed
```

4. مدیر اولیه با رمز `1234` ساخته می‌شود و در اولین ورود ملزم به تغییر رمز است.

## مدل امنیتی

- نام کاربری، کد ملی معتبر و یکتای ۱۰ رقمی است.
- رمزها هرگز به‌صورت خام ذخیره نمی‌شوند و با bcrypt، cost 12، Hash می‌شوند.
- Sessionها با Token تصادفی ۲۵۶ بیتی و Cookie از نوع `HttpOnly`, `SameSite=Lax` نگهداری می‌شوند؛ فقط Hash توکن در PostgreSQL ذخیره می‌شود.
- بعد از پنج ورود ناموفق، حساب ۱۵ دقیقه قفل می‌شود.
- درخواست‌های تغییر‌دهنده دارای کنترل Same-Origin هستند.
- تمام APIهای مدیریت درخواست Reset، نقش و `departmentId` سرپرستار را سمت سرور بررسی می‌کنند.
- کاربری که `mustChangePassword=true` دارد، به‌جز API تغییر رمز به منابع محافظت‌شده دسترسی ندارد.

## مدل مدیر و سرپرستار

در این سامانه هر `HEAD_NURSE` مدیر Tenant یا بخش خودش است. چندین مدیر/سرپرستار مستقل می‌توانند وجود داشته باشند و هر حساب فقط به `departmentId` خودش دسترسی دارد.

سرپرستار جدید از همان صفحه ورود، گزینه «تعریف بخش جدید» را انتخاب می‌کند. `POST /api/onboarding/head-nurse` به‌صورت صریح و create-only بخش خالی را در S3 می‌سازد، حساب سرپرستار را با رمز اولیه `1234` ایجاد می‌کند و او را به همان بخش متصل می‌کند. ساخت اولین بخش به Seed خودکار یا فایل JSON اولیه وابسته نیست.

## ساخت حساب پرسنل

`POST /api/users` فقط برای مدیر سراسری و مدیر/سرپرستار مجاز است. سرپرستار صرفاً می‌تواند کاربر `PERSONNEL` در بخش خودش بسازد. تمام حساب‌های جدید با رمز اولیه `1234`، `mustChangePassword=true` و `hasResetRequest=false` ساخته می‌شوند. ثبت پرسنل در پنل، حساب ورود متناظر را نیز با کد ملی مستقل ایجاد می‌کند.

نمونه:

```json
{
  "nationalId": "0010000003",
  "firstName": "نام",
  "lastName": "نام خانوادگی",
  "role": "PERSONNEL",
  "departmentId": "sepehr",
  "personnelId": "p1"
}
```

`personnelId` شناسه رکورد متناظر در داده S3 است و برای اتصال Session کاربر به داشبورد پرسنلی استفاده می‌شود.

## فراموشی و بازنشانی رمز

1. کاربر کد ملی را در صفحه ورود وارد و «فراموشی رمز عبور» را انتخاب می‌کند.
2. API مقدار `hasResetRequest` را `true` می‌کند، اما پاسخ برای کد ملی موجود و ناموجود یکسان است تا امکان شناسایی کاربران ایجاد نشود.
3. سرپرستار فقط درخواست‌های بخش خودش را می‌بیند و نمایش این فهرست در رابط کاربری به‌طور انحصاری به بخش «مدیریت پرسنل» در پنل مدیر محدود است.
4. اقدام بازنشانی، در یک عملیات سمت سرور:
   - Password Hash را برای `1234` جایگزین می‌کند؛
   - `mustChangePassword=true` می‌کند؛
   - `hasResetRequest=false` و `resetRequestedAt=null` می‌کند؛
   - قفل ورود را پاک می‌کند؛
   - تمام Sessionهای کاربر را باطل می‌کند.

## نکات عملیاتی

- روی APIهای Login، Forgot Password و Onboarding در Load Balancer/WAF نیز Rate Limit توزیع‌شده و CAPTCHA اعمال کنید. اگر ثبت‌نام عمومی مطلوب نیست، مسیر Onboarding را با Invite Code سازمانی محافظت کنید.
- `DATABASE_URL` و اطلاعات مدیر را هرگز Commit نکنید.
- برای کد ملی و اطلاعات پرسنلی Retention policy، Audit log و کنترل دسترسی سازمانی تعریف کنید.
- قبل از Production، کاربران موجود S3 را با کد ملی مستقل به جدول `User` مهاجرت و `personnelId` آنها را متصل کنید.
